After the finding is loaded and before `res.render()`, compute prev/next IDs using the SAME sort order as the student-lab list (difficulty order). Add this block after `localizedFinding.difficulty = DIFFICULTY_MAP[finding.id] || 'medium';` (line 161) and before the `const users` line:

```javascript
// Compute prev/next navigation using difficulty sort order (matches student-lab list)
const allFindings = db.prepare('SELECT id FROM sca_findings').all();
const sortedIds = allFindings
  .map(f => ({ id: f.id, difficulty: DIFFICULTY_MAP[f.id] || 'medium' }))
  .sort((a, b) => DIFFICULTY_ORDER[a.difficulty] - DIFFICULTY_ORDER[b.difficulty])
  .map(f => f.id);
const currentIndex = sortedIds.indexOf(finding.id);
const prevId = currentIndex > 0 ? sortedIds[currentIndex - 1] : null;
const nextId = currentIndex < sortedIds.length - 1 ? sortedIds[currentIndex + 1] : null;
```

Replace the comment on lines 105-108:

```javascript
// -------------------------------------------------------------------------
// CLASSROOM INTERNAL ENDPOINTS (auth-protected since Phase 7 / QWIN-04)
// NOTE: The classroom-manager process (scripts/classroom-manager.js) calls
//       POST /api/instructor-message via plain HTTP without a session cookie.
//       After this change, classroom-manager broadcasts will receive a redirect
```

**Important context on classroom-manager impact:** The classroom-manager's `broadcastToInstance()` function (scripts/classroom-manager.js line 184) makes HTTP POST requests to `/api/instructor-message` without a session cookie. Adding `requireAuth` means these requests will be redirected to the login page instead of setting the broadcast message. This is the correct behavior per QWIN-04 ("POST /api/instructor-message requires authentication"). The classroom-manager broadcast feature was designed for an unauthenticated internal network; the product review explicitly flagged this as a security gap to close. If broadcasts are needed later, the classroom-manager can be updated to authenticate first.

The GET `/api/instructor-message` endpoint is polled by the header banner script (header.ejs line ~432). Since header.ejs is only included for authenticated users, the session cookie is sent automatically via same-origin fetch. The existing `catch(() => {})` error handler on line ~444 handles any edge cases gracefully.
